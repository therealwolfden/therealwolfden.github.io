Hash Tables: Collision Resolution

Collision Resolution Strategies

The Problem where position h(k) is occupied by key k
  • We can store the solution elsewhere in the hash table structure.
  • Two-level data structure.
    • Hash table serves as the index and divides the map into buckets.
    • Buckets are a secondary map within the hash table.

Chaining Strategies

Seperate Chaining
  • Items are not stored in the hash table itself but rather a secondary map.
  • Insert operations to the correct position.
  • Performance:
    • Depends on the load factor (λ).
      • λ = n / m where n is the number of buckets and m is the capacity in the hash table array.
    • Often stated in terms of the number of probes or accesses to the data structure.
      • S(λ) is the average number of probes required for a successful search.
      • U(λ) is the average number of probes required for an unsuccessful search.
      • Search requires (1 + n) probes if the secondary map is an unordered list O(n) when all values hash to the same value.
      • Search requires (1 + logn) probes if the secondary map is a balanced search tree O(logn) when all values hash to the same value.
  • Advantages:
    • Easy to implement.
    • Provides good average runtime with a good hash function.
  • Disadvantages:
    • Requires a second data structure.

Coalesced Chaining
  • Stores collisions in the next space of the hash table.
  • If moving to the next space causes a collision, repeat.
  • Each value in the table needs to store a next value.
  • Sometimes a "cellar" is implemented to resolve collisions.
    • Roughly 14% of the hash table array is reserved for collisions.
    • If the "cellar" fails, begin to populate the normal section of the array.
  • Advantages:
    • Does not require a second data structure.
    • Provides good average runtime with a good hash function.
  • Disadvantages:
    • Harder to implement, especially when removing a key.

Open Addressing Strategies

Overview
  • Items are stored directly in the hash table.
  • If collisions occur the position is guided through a secondary hash function.

Linear Probing
  • Place colliding item in the next (circularly) available table position
  • H(k, i) = (h(k) + i) mod m
  • Performance:
    • S(λ) = 1⁄2(1 + 1⁄1 - λ)
    • U(λ) = 1⁄2(1 + 1⁄(1 - λ)2)
  • Advantages:
    • Easy to implement.
    • Satisfactory performance when table not too full (λ < .5)
  • Disadvantages:
    • Primary clustering slows down searches and creates more collisions.
    • Primary clustering grows as more collisions occur at the same location.

Double Hashing
  • Uses a secondary hash function which can not equal 0.
  • H(k, i) = (h(k) + i*h2(k)) mod m
  • Performance:
    • S(λ) = 1⁄λln(1⁄1 - λ)
    • U(λ) = 1⁄λln(1⁄(1 - λ)2)
  • Advantages:
    • Easy to implement.
    • Breaks up clusters which shortens expected runtime.

Rehashing

Resizing
  • As the loading factor (λ) increases the performance of the hash table becomes inefficient.
  • When a load factor gets to high we need to rehash or resize the hash table.
  • To rehash:
    1. Create a new larger array.
    2. Scan through the old array and put each entry into the new array.
Algorithm resize(A)
Input an array A
Output a new hash table
// Create a larger array
h ← new array of size length(A) * 2 + 1
// Rehash
for each entry e in A
put(H, e)
return H

Deleting

Deleting
  • Seperate Chaining: easy
  • Coalesced Chaining and Open Addressing: complicated as each field now needs to store a isDeleted parameter.
    • Treat isDeleted == true as empty buckets.
    • It is helpful to rehash and ignore deleted entries in the rehashed table.
